# Starseekers

예전에 자기가 별표한 GitHub 저장소들을 의미 기반으로 검색하는 로컬 CLI 툴을 만들어 배포한 적이 있다.
사실 초기 기획은 웹 사이트로 해서 여러 사람이 (사실 그냥 내가) 쉽게 쓸 수 있게 하는 거였는데 너무 귀찮아서 그런 것.
그 이후로 7달이 지났고 저번 글에서 ChatGPT를 구독해서 Codex를 쓸 수 있게 되었다 했다.
마침 오늘 하루종일 버스를 탔어야 했어서 폰으로 Codex Web을 통해 바이브 코딩을 시도해봤고 꽤 성공적이었다.

GitHub에서 저장소를 만들고 Codex Web에서 예전에 작성했던 아이디어 노트를 주며 이대로 구현해달라 했다.
처음부터 그럴싸하게 코드가 나왔지만 빌드가 안 됐다.
그래도 빌드 될 때까지 고쳐달라고 하니 고쳐준다.

Vercel에서 배포해보니 그럴싸한 UI가 나왔지만 막상 API 호출이 제대로 동작을 안 했다.
런타임 오류 복붙 해주면서 고쳐달라고 몇번 하니 의도한 동작을 하게 됐다.

디자인이 마음에 안 들어서 ChatGPT에게 이러이러한 서비스를 만들테니 프론트엔드 디자인을 코드와 함께 해달라고 했다.
나온 거 그대로 Codex에 붙혀 넣고 해달라 하니 모-던한 UI가 나왔다.

이후로도 다른 사람에게 공개 가능한 최소한의 수준이 되게끔 십수번의 대화를 주고 받았다.
이렇게 주고 받으면서 점점 원하는 결과물이 나오는 게 참 성능이 많이 좋아졌구나 느꼈다.

주니어~시니어 사이 어딘가의 개발자들을 옆에 두고 입코딩으로 시키는 느낌이었다.

이렇게 해보면서 말로만 들었던 바이브코딩의 장단점이 확실히 피부로 와닿았다.
효율은 측정 안 해봐서 모르겠고 확실히 편하긴 하다. 말만 하면 다 해주니까. 그것도 병렬로.
다만 작성된 코드가 눈에 안 들어온다. 남이 짠 코드니까 당연하긴 한데 오래 유지보수 해야하는 프로젝트에서 이러면 꽤 곤란한 일이 많을거다.
나는 변경 사항들을 리뷰했는데도 이 정도면....

그리고 말로 시키는 게 의외로 어려운 일이란 것도 알았다. 내가 생각하고 있는 걸 100% 말로 표현해줄 수 없기 때문에 결과물에 항상 피드백이 생긴다. AI는 거의 시킨대로만 해줘서 사람이 쌓아올린 노하우가 자연스럽게 녹아들길 기대하긴 좀 이른 거 같다.
대량의 데이터를 처리해야하는 부분인데 해달라고 하기 전까지 배치 처리를 전혀 안 한다거나...

아무튼 값진 경험이었다.
