# AI 서비스의 응답 소요 시간 예측은 어떻게 할까

> AI 서비스의 응답 소요 시간을 예측하는 문제는 단순한 카운트다운 문제가 아니다. 단일 LLM 호출만 놓고 보면 핵심은 토큰 생성 속도보다 **앞으로 얼마나 더 생성될지**, 즉 남은 출력 길이를 어떻게 추정하느냐에 있다. 최근 연구는 이를 고정된 한 값보다 **분포**로 보고, 프롬프트 표면 feature보다 **hidden state 기반**, 호출 시작 시 한 번 예측하는 방식보다 **생성 중 재예측**하는 방향으로 가고 있다. 한편 검색, 도구 호출, 여러 번의 LLM 호출이 섞인 에이전틱 서비스에서는 문제가 다시 커진다. 이 경우 단일 LLM 호출의 길이 예측은 전체의 한 부품일 뿐이며, 실제로는 **현재 단계, 남은 작업 구조, 각 단계의 실행시간 분포**를 함께 봐야 전체 응답 시간을 더 자연스럽게 설명할 수 있다.

AI 서비스의 응답 소요 시간을 예측하는 문제를 조사해보면, 생각보다 층위가 여러 개라는 점이 먼저 보인다.

겉으로는 단순히 “답변이 언제 끝나는가”처럼 보이지만, 실제로는 다음이 섞여 있다.

- LLM 한 번 호출의 종료 시점 예측
    
- 스트리밍 중 남은 출력 길이 예측
    
- 검색, 도구 호출, 재시도를 포함한 에이전틱 워크플로우 전체 시간 예측
    
- 사용자에게 이 값을 어떤 형태로 보여주는가의 문제
    

이 글은 이 문제를 바로 적용 가능한 방법으로 정리한 문서가 아니다.  
최근에 관련 아이디어와 연구 방향을 간단히 조사해보며, **AI 서비스의 응답 소요 시간 예측을 어떤 문제로 봐야 하는지**를 정리한 메모에 가깝다.

## 단일 LLM 호출부터 보는 것이 자연스럽다

가장 기초가 되는 경우는 LLM 한 번 호출하고, 그 출력이 끝나면 응답이 완료되는 구조다.

이 경우 응답 시간은 대략 두 부분으로 나눌 수 있다.

- 첫 토큰이 나오기 전까지의 시간
    
- 첫 토큰 이후 나머지 토큰이 생성되는 시간
    

운영 중인 서비스에서는 토큰 생성 속도 자체는 비교적 쉽게 계측할 수 있다.  
그래서 실제로 더 어려운 문제는 **앞으로 토큰이 얼마나 더 나올지**를 추정하는 쪽이다.

즉 단일 LLM 호출의 응답 시간 예측은 결국 이런 질문으로 좁혀진다.

- 이번 호출은 최종적으로 얼마나 길게 끝날까
    
- 지금 이 시점에서 앞으로 몇 토큰 정도 더 남았을까
    
- 이 값을 하나의 숫자가 아니라 범위나 분포로 볼 수 있을까
    

이 관점에서 보면, 응답 시간 예측은 사실상 **출력 길이 예측 문제**와 아주 강하게 연결된다.

## 출력 길이를 하나의 값으로 보는 것은 잘 맞지 않는다

이 문제를 조사해보며 가장 먼저 눈에 띄는 점은, 출력 길이를 하나의 고정된 정답처럼 다루기 어렵다는 것이다.

LLM은 확률적으로 다음 토큰을 선택한다.  
같은 프롬프트라도 결과 길이는 항상 같지 않다.  
또 실제 서비스에서는 다음 요소들이 길이에 영향을 준다.

- temperature
    
- top-p
    
- top-k
    
- repetition penalty
    
- stop sequence
    
- max tokens
    

그래서 “이 프롬프트의 정답 길이는 몇 토큰이다”처럼 생각하면 설명이 잘 안 된다.  
오히려 더 자연스러운 관점은 이렇다.

- 길이는 고정값이 아니라 확률변수다
    
- 예측 대상은 하나의 숫자보다 길이의 분포다
    
- ETA도 단일 숫자보다 범위 형태가 더 자연스럽다
    

예를 들어 이런 식이다.

- 보통은 이 정도에서 끝난다
    
- 조금 길어지면 이 정도까지 간다
    
- 길게 끌리면 이 정도까지도 갈 수 있다
    

즉 “정확히 몇 토큰 남았는가”보다 **얼마나 남았을 가능성이 높은가**가 더 중요한 문제가 된다.

## 길이 예측 연구는 무엇을 하려는가

관련 연구를 보면, 이 문제를 아주 단순하게 보지는 않는다.  
대체로 두 가지 축이 보인다.

첫째는 **총 출력 길이 예측**이다.  
호출 시작 시점에 “이번 응답은 대략 어느 정도 길이일까”를 추정한다.

둘째는 **남은 출력 길이 예측**이다.  
이미 일부 토큰이 생성된 상태에서 “앞으로 얼마나 더 남았을까”를 다시 추정한다.

실제 응답 시간 예측에 더 직접적으로 맞닿아 있는 것은 둘째다.  
호출 시작 전에 한 번 대충 맞히는 것보다, 스트리밍이 진행되는 동안 현재 상태를 반영해 남은 길이를 다시 보는 편이 훨씬 자연스럽기 때문이다.

이 점에서 최근 방향은 대체로 다음처럼 보인다.

- 정적 총길이 예측보다 동적 남은길이 예측
    
- 표면 feature보다 모델 내부 표현 활용
    
- 점추정보다 분포 예측
    

## hidden state 기반 접근이 눈에 띈다

최근 쪽으로 갈수록 눈에 띄는 것은 **hidden state를 활용하는 접근**이다.

처음에는 predictor 입력으로 이런 것들을 떠올리기 쉽다.

- 프롬프트 길이
    
- 프롬프트 내용
    
- 현재까지 생성된 토큰
    
- 태스크 유형
    
- 모델 종류
    

하지만 hidden state를 쓰는 쪽으로 가면 생각이 조금 달라진다.

프롬프트와 지금까지 생성한 토큰은 이미 모델 내부 상태에 반영되어 있다고 볼 수 있다.  
즉 현재 시점의 hidden state는 “지금까지 어떤 입력이 들어왔고 어떤 출력이 생성되었는가”를 압축한 표현처럼 다룰 수 있다.

이 관점이 매력적인 이유는 분명하다.

- 사람이 임의로 만든 feature에 덜 의존할 수 있다
    
- 현재까지의 생성 상태를 자연스럽게 반영할 수 있다
    
- 정적 예측뿐 아니라 진행 중 재예측에도 잘 맞는다
    

조사해보면 최근 연구도 별도 큰 보조 모델을 두는 것보다, **메인 모델의 내부 표현을 재사용해 길이를 예측하는 방향**을 꽤 강하게 밀고 있다.

## 그렇다고 모든 변수를 hidden state 하나로 덮을 수는 없다

다만 hidden state를 쓴다고 해서 모든 변수를 다 지울 수 있는 것은 아니다.

프롬프트와 지금까지의 출력은 hidden state에 반영된다고 볼 수 있다.  
출력 종류 역시 어느 정도는 입력과 현재 상태 속에 녹아 있다고 볼 수 있다.

하지만 디코딩 설정은 다르다.

- temperature
    
- top-p
    
- top-k
    
- repetition penalty
    
- stop 조건
    

이런 값들은 “지금까지 무엇이 생성되었는가”와는 별개로, **앞으로 어떻게 생성할 것인가**를 바꾸는 외생 조건에 가깝다.  
그래서 hidden state만으로 모두 대체하기보다, 길이 예측기의 조건변수로 따로 넣는 쪽이 더 자연스러워 보인다.

## 모델은 feature라기보다 predictor 자체를 나누는 축에 가깝다

조사하며 따져보면 모델 자체도 흥미로운 변수다.

처음에는 model id를 predictor의 feature 하나로 넣는 방식을 떠올릴 수 있다.  
하지만 hidden state 기반 predictor를 상정하면, 이보다는 **모델별로 predictor를 따로 두는 구조**가 더 자연스럽다.

이유는 단순하다.

- hidden state 공간 자체가 모델마다 다르다
    
- 레이어별 표현 특성이 다르다
    
- verbosity 경향이 다르다
    
- EOS 성향과 stop behavior가 다르다
    
- 샘플링 파라미터에 대한 민감도도 다르다
    

그래서 이 문제는 “공용 predictor + model id feature”라기보다, **각 모델에 붙는 전용 predictor 또는 전용 head**에 더 가깝게 느껴진다.

즉 입력 변수의 일부로 모델을 넣기보다는, 모델이 바뀌면 예측기 자체도 달라지는 구조가 더 자연스럽다.

## 최신 성능은 어느 정도인가

여기서 기대치를 너무 높게 잡으면 안 된다.  
최신 SOTA 계열도 “정확히 언제 끝날지”를 정밀하게 맞히는 수준은 아니다.

조사해보면 최근 논문들은 대체로 다음 정도의 인상을 준다.

- 기존 baseline보다 의미 있게 좋아진다
    
- 하지만 절대 오차는 여전히 작지 않다
    
- 수십 토큰대에서 100토큰 안팎의 오차가 남는 경우도 흔하다
    
- 즉 정밀한 카운트다운보다는 구간형 예측과 스케줄링 최적화에 더 적합하다
    

이건 오히려 자연스러운 결과처럼 보인다.  
예측 대상 자체가 확률적이기 때문이다.

같은 프롬프트라도 한 번은 짧게 끝나고, 다른 한 번은 설명이 좀 더 붙을 수 있다.  
그래서 최근 방향은 단일 숫자 회귀 하나보다, **분위수 예측이나 분포 예측** 쪽으로 가는 흐름이 강하다.

즉 이 기술은 “정확히 7.3초 남았다”를 맞히기 위한 기술이라기보다,  
“짧게 끝날 호출인가, 길어질 가능성이 큰 호출인가”를 구분하고,  
스트리밍이 진행될수록 그 범위를 점점 좁혀가는 기술에 더 가깝다.

## 단일 LLM 호출 기준으로 보면 무엇이 핵심인가

여기까지 조사해보며 정리하면, 단일 LLM 호출의 응답 소요 시간 예측에서 핵심은 대략 다음처럼 보인다.

첫째, 응답 시간 예측은 결국 **출력 길이 예측** 문제와 연결된다.

둘째, 출력 길이는 고정값이 아니라 **분포**로 보는 편이 더 자연스럽다.

셋째, 최근에는 프롬프트 표면 feature보다 **hidden state 기반 접근**이 더 유력해 보인다.

넷째, 호출 시작 전에 한 번 예측하는 것보다, **진행 중 남은 길이를 다시 예측하는 방식**이 실용적이다.

다섯째, 모델은 공용 predictor의 feature 하나라기보다 **모델별 predictor를 분리해야 하는 축**에 가깝다.

여섯째, 디코딩 설정은 hidden state 밖의 조건으로 별도 취급하는 편이 더 맞아 보인다.

이 정도가 현재 시점에서 가장 일관된 그림처럼 느껴진다.

## 그런데 실제 서비스는 여기서 끝나지 않는다

문제는 요즘 AI 서비스가 단일 LLM 호출로 끝나는 경우가 점점 줄고 있다는 점이다.

실제 서비스는 흔히 이런 식으로 움직인다.

1. planner 역할의 호출이 계획을 세운다
    
2. 검색 또는 데이터 조회 도구를 호출한다
    
3. 그 결과를 다시 읽고 중간 정리용 LLM 호출을 한다
    
4. 필요하면 추가 도구를 호출한다
    
5. 검증하거나 재시도한다
    
6. 마지막으로 최종 답변용 LLM 호출을 한다
    

이 구조에서는 전체 응답 시간이 더 이상 “이번 LLM 호출이 몇 토큰 남았는가”만으로 설명되지 않는다.

단일 호출 예측은 여전히 필요하다.  
하지만 전체 서비스 응답 시간은 이제 **워크플로우 전체의 남은 실행시간 예측** 문제가 된다.

즉 문제의 성격 자체가 바뀐다.

## 에이전틱 서비스에서는 무엇이 달라지나

에이전틱 서비스의 응답 소요 시간은 대략 다음의 합으로 볼 수 있다.

- 오케스트레이션 시간
    
- 각 LLM 호출 시간
    
- 각 도구 실행 시간
    
- 외부 API 대기 시간
    
- 병렬 작업 합류 시간
    
- 재시도와 fallback 시간
    

이 중 단일 LLM 호출 시간은 전체의 일부일 뿐이다.  
그리고 나머지 항목들은 토큰 길이 예측으로는 설명되지 않는다.

예를 들어 이런 상황을 생각해볼 수 있다.

- 첫 번째 LLM 호출은 짧게 끝났지만, 검색 결과가 많아 후속 단계가 길어진다
    
- 병렬 작업 두 개 중 하나가 느려서 전체가 지연된다
    
- 외부 API 재시도가 발생해 tail latency가 커진다
    
- 첫 번째 계획 단계에서는 짧게 끝날 것 같았는데, 중간에 추가 탐색이 늘어난다
    

이 경우 전체 ETA는 본질적으로 **현재 workflow state를 조건으로 한 남은 시간 분포** 문제다.

즉 단일 LLM 호출에서는 “남은 출력 길이”가 핵심 상태였다면,  
에이전틱 서비스에서는 “현재 어떤 단계에 있고, 무엇이 끝났고, 무엇이 남았는가”가 핵심 상태가 된다.

## 그래서 사용자에게 무엇을 보여주는 게 좋은가

에이전틱 서비스에서 가장 먼저 든 생각은 “전체 ETA를 하나의 숫자로 보여주면 되지 않나”일 수 있다.  
하지만 조사해보면 이 방식은 불안정할 가능성이 높다.

에이전틱 시스템은 다음 이유로 변동성이 크다.

- 단계 수가 고정되지 않을 수 있다
    
- 도중에 추가 호출이 생길 수 있다
    
- 도구 실패로 재시도할 수 있다
    
- 병렬 작업 중 느린 꼬리가 전체를 결정할 수 있다
    

그래서 이 경우 사용자에게 유용한 정보는 단일 숫자 하나보다 **계층형 진행 상태**에 더 가깝다.

### 1. 지금 무엇을 하고 있는지

예:

- 계획 수립 중
    
- 자료 수집 중
    
- 도구 실행 중
    
- 결과 검증 중
    
- 최종 정리 중
    

이 정보는 아주 중요하다.  
사용자는 “정확히 몇 초 남았는가”보다 “지금 멈춰 있는 것이 아니라 실제로 무언가 진행 중인가”를 먼저 알고 싶어하기 때문이다.

### 2. 남은 작업 구조가 어떻게 되는지

예:

- 5단계 중 2단계 완료
    
- 검색 4개 중 3개 완료
    
- 병렬 하위 작업 3개 중 2개 완료
    
- 최종 답변 생성만 남음
    

에이전틱 서비스에서는 시간 그 자체보다 **남은 일의 구조**가 더 신뢰할 만한 정보가 되는 경우가 많다.

### 3. ETA는 범위로만 보여주기

예:

- 보통 10~20초
    
- 거의 완료
    
- 추가 확인 중이라 조금 더 걸릴 수 있음
    

단일 숫자는 자주 틀릴 수 있지만, 범위형 ETA는 훨씬 견고하다.  
특히 도구 호출과 외부 API가 섞인 구조에서는 이 차이가 더 크게 난다.

## 내부적으로는 어떻게 생각해야 하나

단일 LLM 호출의 경우는 대략 이렇게 생각할 수 있다.

- 남은 토큰 수의 분포
    
- 곱하기
    
- 현재 디코딩 속도
    

반면 에이전틱 서비스 전체는 이렇게 생각하는 편이 더 자연스럽다.

- 현재 남은 노드들의 실행시간 분포
    
- 의존관계에 따라 직렬 또는 병렬로 합성
    
- 재시도와 분기 가능성을 반영
    

즉 직렬 단계는 더하고, 병렬 단계는 최대값에 가깝게 보고, 분기 단계는 기대값이나 보수적 분위수로 보는 식이다.

이렇게 보면 에이전틱 ETA는 **토큰 예측 문제**라기보다 **워크플로우 런타임 예측 문제**에 가깝다.

그리고 여기서 단일 LLM 호출의 길이 예측기는 전체 시스템을 구성하는 한 노드의 predictor가 된다.

## 에이전틱 서비스에서는 왜 설명형 피드백이 더 중요해지나

단일 LLM 호출에서는 어느 정도 “남은 시간” 자체가 핵심이 될 수 있다.  
하지만 에이전틱 서비스에서는 시간이 길어질수록, 사용자 입장에서는 숫자보다 **설명 가능한 진행 상태**가 더 중요해진다.

예를 들어 다음 둘 중 무엇이 더 나은지는 분명하다.

- 27초 남음
    
- 검색 완료, 문서 2개 분석 완료, 최종 요약 생성 중
    

후자가 더 유용하다.  
왜 오래 걸리는지, 어디까지 진행됐는지, 멈춘 것은 아닌지 알 수 있기 때문이다.

즉 에이전틱 서비스에서는 ETA를 숫자로만 주는 것보다,  
**현재 단계 + 남은 작업 구조 + 구간형 ETA**의 조합이 더 자연스럽다.

## 조사해보며 든 결론

이 문제를 조사해보며 든 결론은 비교적 분명하다.

단일 LLM 호출의 응답 소요 시간 예측은 결국 **남은 출력 길이 예측** 문제에 가깝다.  
그리고 최근 방향은 hidden state 기반, 동적 재예측, 분포 예측 쪽으로 모인다.

반면 에이전틱 서비스의 응답 소요 시간 예측은 **워크플로우 전체의 남은 실행시간 예측** 문제에 가깝다.  
이 경우 단일 LLM 호출 예측은 전체의 한 부품이 된다.

그래서 “AI 서비스의 응답 소요 시간 예측”이라는 제목 아래에서 보면, 문제는 두 층으로 나뉜다.

- 아래층: 단일 LLM 호출의 남은 길이 예측
    
- 위층: 에이전틱 워크플로우 전체의 남은 시간 예측
    

현재까지 조사해본 인상으로는,  
아래층은 이미 꽤 직접적으로 연구가 진행되고 있고,  
위층은 아직 더 시스템적인 문제로 남아 있다.

즉 오늘날 AI 서비스의 응답 소요 시간 예측은,  
단순히 “몇 초 남았는가”를 맞히는 문제가 아니라  
**현재 상태를 기준으로 앞으로 남은 생성량과 남은 작업량을 어떤 수준으로 신뢰성 있게 추정할 것인가**의 문제라고 보는 편이 가장 자연스러워 보인다.
